Examples

DataTalksClub contact lifecycle

curl -sS -X POST "$DATAMAILER_URL/api/contacts" \
  -H "Authorization: Bearer $DATAMAILER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "learner@example.com",
    "audience": "datatalksclub",
    "client": "dtc-courses",
    "status": "subscribed",
    "tags": ["ml-zoomcamp"],
    "verified": false
  }'

curl -sS "$DATAMAILER_URL/api/contacts/status?email=learner@example.com&audience=datatalksclub&client=dtc-courses" \
  -H "Authorization: Bearer $DATAMAILER_API_KEY"

curl -sS -X PATCH "$DATAMAILER_URL/api/contacts/{contact_id}/verification" \
  -H "Authorization: Bearer $DATAMAILER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"audience":"datatalksclub","client":"dtc-courses","verified":true}'

curl -sS "$DATAMAILER_URL/api/contacts?audience=datatalksclub&client=dtc-courses&tags=ml-zoomcamp&limit=100" \
  -H "Authorization: Bearer $DATAMAILER_API_KEY"

Transactional registration, reset, and verification

import requests

base_url = "http://localhost:8000"
headers = {"Authorization": f"Bearer {datamailer_api_key}"}

requests.post(
    f"{base_url}/api/transactional/send",
    headers=headers,
    json={
        "email": "user@example.com",
        "template_key": "registration-welcome",
        "idempotency_key": "registration-user-123",
        "context": {"name": "User", "course_name": "ML Zoomcamp"},
        "metadata": {"source": "registration"},
    },
    timeout=10,
)

requests.post(
    f"{base_url}/api/transactional/send",
    headers=headers,
    json={
        "email": "user@example.com",
        "template_key": "password-reset",
        "idempotency_key": "password-reset-user-123-request-456",
        "context": {"reset_url": "https://client.example/reset/placeholder"},
        "metadata": {"source": "password-reset"},
    },
    timeout=10,
)

requests.post(
    f"{base_url}/api/transactional/send",
    headers=headers,
    json={
        "email": "user@example.com",
        "template_key": "email-verification",
        "idempotency_key": "verify-user-123-email-1",
        "context": {"name": "User", "verification_url": "https://client.example/verify/placeholder"},
        "metadata": {"source": "email-verification"},
    },
    timeout=10,
)

# After the user verifies in the client app, mark the Datamailer contact verified.
requests.patch(
    f"{base_url}/api/contacts/{contact_id}/verification",
    headers=headers,
    json={
        "audience": "datatalksclub",
        "client": "dtc-courses",
        "verified": True,
    },
    timeout=10,
)

requests.post(
    f"{base_url}/api/contacts/imports",
    headers=headers,
    json={
        "audience": "ai-shipping-labs",
        "client": "ai-shipping-labs",
        "dry_run": False,
        "contacts": [
            {
                "email": "subscriber@example.com",
                "status": "subscribed",
                "tags": ["newsletter"],
                "email_validation": {"status": "externally_validated"},
            }
        ],
    },
    timeout=10,
)

requests.get(
    f"{base_url}/api/contacts.csv",
    headers=headers,
    params={"audience": "ai-shipping-labs", "client": "ai-shipping-labs"},
    timeout=10,
)

Notes

Transactional templates are visible to staff in Templates. Client apps generate verification and reset URLs, pass them as context, and complete verification in their own app before marking the Datamailer contact verified.

Hard-bounced or complained contacts are blocked for transactional sends. Missing required template context returns a JSON validation error before Datamailer creates a contact, message, event, or queue payload.

Public tracking, unsubscribe, and SES webhook routes are documented for integration context; client application endpoints under /api use Bearer authentication.
